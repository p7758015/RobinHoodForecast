# RobinHoodForecast v2 — Blueprint логики прогнозирования матчей чемпионата

## 1. Назначение документа

Этот blueprint фиксирует рабочую v1-спецификацию логики прогнозирования **матчей чемпионата** для RobinHoodForecast v2.  
Он нужен как источник истины перед реализацией в коде и как база, на которой потом будут собраны логики для кубков и международных турниров.

Ключевой принцип проекта:

- LLM / OpenClaw **не прогнозирует исход**.
- OpenClaw используется как **агент-сборщик и нормализатор информации**.
- Прогноз рассчитывается **только кодом** через явные факторы, веса, правила и фильтры.
- Для каждого матча должен формироваться **структурированный снимок контекста матча**, который сохраняется в БД и может обновляться по мере появления новой информации.

---

## 2. Целевой пользовательский сценарий

На вход система должна принимать следующие типы запросов:

### 2.1. Прогноз на конкретный матч
Примеры:
- `дай прогноз на Реал Мадрид — Валенсия`
- `прогноз на Арсенал — Челси 2026-05-21`

### 2.2. Прогноз на несколько матчей
Примеры:
- `дай прогноз на Барселона — Севилья и Интер — Милан`
- `проанализируй весь следующий тур Серии А`

### 2.3. Прогноз на тур / дату / диапазон дат
Примеры:
- `дай прогноз на весь следующий тур АПЛ`
- `проанализируй матчи 2026-05-21`
- `дай прогнозы на матчи 21 и 22 мая`

### 2.4. Сбор экспресса
Примеры:
- `собери экспресс на 3.5 на завтра`
- `собери экспресс на кэф 5 на 21 и 22 мая`

Правила режима экспресса:
- экспресс собирается по **лигам/турнирам, явно заданным пользователем** (дата, лига, матчи) и прошедшим **registry/config allow-list** (`league_registry` + `LEAGUE_EXPRESS_ALLOWED_CODES` / `LEAGUE_DENY_CODES`);
- жёсткий фильтр «только топ-5 чемпионатов Европы» **снят** (Stage 3): non-top-5 лиги (например Botola Pro / `FS_BOTOLA_PRO`) допускаются, если разрешены в registry или env;
- неизвестные коды соревнований не ломают pipeline: для express — soft skip + WARNING; для анализа — defaults + WARNING;
- selection policy для экспресса должна быть заметно строже, чем для обычного прогноза (scorer `express_safety`, odds/probability thresholds);
- матчи с повышенным календарным, ротационным и составным риском должны штрафоваться или исключаться.

---

## 3. Базовый архитектурный принцип

Система должна работать по схеме:

1. Парсинг запроса пользователя.
2. Определение типа запроса и списка матчей-кандидатов.
3. Сбор и нормализация данных по каждому матчу.
4. Сохранение снимка данных в БД.
5. Расчёт факторов и итогового прогноза кодом.
6. Формирование ответа пользователю.
7. Для сыгранных матчей — закрытие результатов и накопление статистики точности.

### Ключевое разделение ролей

#### OpenClaw / агент
Используется для:
- поиска информации;
- чтения и извлечения фактов;
- нормализации фактов в JSON-схему;
- обновления данных по тренерам, травмам, составам, новостям, календарю.

#### Python-код
Используется для:
- расчёта факторов;
- классификации типа матча;
- расчёта вероятностей и рисков;
- отбора событий для экспресса;
- учёта статистики качества модели.

---

## 4. Что должно храниться в БД

Система должна хранить не только итоговые прогнозы, но и **структурированные данные по футбольному контексту**.

### 4.1. Медленные данные
Обновляются редко:
- турниры;
- команды;
- профили тренеров;
- история назначений тренеров;
- игроки и профили игроков;
- базовый календарь турниров;
- структура сезона.

### 4.2. Средние данные
Обновляются регулярно:
- турнирная таблица;
- календарь команды по всем турнирам;
- последние матчи;
- H2H команд;
- история тренерских противостояний;
- сила и роль игроков;
- статус тренера в команде.

### 4.3. Быстрые данные
Обновляются ближе к матчу:
- ожидаемые составы;
- травмы;
- дисквалификации;
- doubtful;
- odds;
- news_context;
- факт смены тренера;
- переносы;
- neutral venue;
- подтверждение ротации.

### 4.4. Итоговый снимок матча
Для каждого анализируемого матча должна храниться сущность типа `match_analysis_snapshot`, содержащая:
- raw collected data;
- normalized data;
- feature values;
- итоговый score;
- confidence;
- timestamp последнего обновления;
- статус свежести данных.

## 7. Главные принципы league logic

### 7.1. Прогноз строится на матч, а не на абстрактную силу команды
То есть базовая сила команды — только один слой из нескольких.

### 7.2. Форма и H2H — не одно и то же
- форма команды = её собственные последние матчи;
- H2H = слабый contextual factor.

### 7.3. Поздняя стадия сезона усиливает турнирную мотивацию
Чем меньше туров осталось, тем больше вес:
- таблицы;
- математической цели;
- очковой дистанции;
- стоимости именно этого матча.

### 7.4. Смена тренера — это special mode
- первый матч нового тренера = отдельный high-impact case;
- матчи 2–4 = окно повышенного тренерского импульса;
- затем влияние стабилизируется.

### 7.5. Календарь по всем турнирам обязателен
Оценка матча чемпионата без анализа соседних матчей команды по еврокубкам / кубку / лиге — неполная и потенциально опасная.

### 7.6. Для экспресса матч должен быть не просто “хорошим”, а “безопасным”
Поэтому часть матчей, которые допустимы для одиночного анализа, может быть запрещена для экспресса.

## 8. Блоки факторов для матча чемпионата

Итоговая модель должна считать матч через следующие факторы:

1. `baseline_strength`
2. `current_form`
3. `motivation`
4. `squad_availability`
5. `coach_factor`
6. `schedule_context`
7. `h2h_context_bias`

---

## 9. baseline_strength

Это долгосрочная сила команды, не зависящая полностью от текущего матча.

В базовый слой можно включать:
- место в таблице;
- очки;
- goal difference;
- длинную форму;
- качество состава;
- средний уровень команды в сезоне;
- домашний/выездной baseline.

Важно:
- baseline не должен доминировать над матчевым контекстом;
- фаворит не должен автоматически получать “надёжный” прогноз только из-за имени.

---

## 10. current_form

Форма должна считаться как смесь нескольких компонентов:

- `recent_form_last_5`
- `recent_form_last_10`
- `home_away_form`
- `form_under_current_coach`
- `performance_trend`

### Рекомендация по структуре form score
`current_form` = смесь:
- долгосрочной формы;
- последних 5 матчей;
- формы при текущем тренере;
- split по home/away.

### Правила
- если тренер сменился недавно, форма при текущем тренере получает повышенный вес;
- если у тренера только 1 матч, это не должно полностью перезаписывать сезонную форму;
- если у команды сильный тренд вверх/вниз, он должен учитываться отдельно.

---

## 11. motivation

Мотивация — это **турнирный смысл** текущего матча.

Она должна зависеть от:
- положения в таблице;
- плотности таблицы;
- количества туров до конца;
- математической достижимости целей;
- зоны борьбы (титул, еврокубки, выживание);
- отсутствия турнирной цели;
- важности конкретного матча для этой цели.

### Примеры
- команда борется за выживание за 3 тура до конца → очень высокая мотивация;
- команда в середине таблицы без шансов вверх/вниз → низкая мотивация;
- команда борется за ЛЧ → высокая мотивация.

### Важно
Календарный фактор **не входит напрямую** в motivation.  
Он идёт отдельным блоком, иначе модель будет путать:
- “матч важен”
и
- “команда может сыграть неосновой / экономить силы”.

---

## 12. squad_availability

Должен оценивать не число отсутствующих, а **важность отсутствий**.

### Что учитывать
- отсутствие вратаря №1;
- отсутствие ключевого ЦЗ;
- отсутствие опорного игрока;
- отсутствие главного креативного игрока;
- отсутствие лучшего бомбардира;
- число missing players;
- число critically missing players;
- стабильность ожидаемой стартовой XI.

### Итоговые подфакторы
- `availability_score`
- `key_absence_impact_score`
- `line_stability_score`
- `lineup_confidence_score`

---

## 13. coach_factor

Coach factor должен состоять из трёх частей:

### 13.1. Базовая сила тренера
- win rate;
- устойчивость результатов;
- сила на дистанции.

### 13.2. История против соперника
- coach vs opponent team;
- coach vs opponent coach.

### 13.3. Контекст текущего назначения
- сколько дней в команде;
- сколько матчей в команде;
- первый ли это матч;
- входит ли тренер в окно “new coach bounce”.

### Правила
- матч 1 нового тренера → максимальный special boost;
- матчи 2–4 → умеренно повышенное влияние;
- далее — постепенная нормализация.

---

## 14. schedule_context

Это один из ключевых блоков v2.

### 14.1. Почему он нужен
Топ-клубы и участники нескольких турниров часто:
- ротируют состав перед важным матчем;
- снижают интенсивность перед еврокубком;
- эмоционально и физически проседают после топ-матча;
- расслабляются после крупной победы или важной еврокубковой игры.

Поэтому календарь должен учитываться отдельно и явно.

### 14.2. Что анализируется
Для каждой команды анализируется окно:
- предыдущий матч;
- следующий матч;
- последние 14 дней;
- ближайшие 7 дней.

### 14.3. Ключевые поля
- `days_since_last_match`
- `days_to_next_match`
- `matches_last_14_days`
- `matches_next_7_days`
- `prev_match_importance`
- `next_match_importance`
- `prev_match_opponent_strength`
- `next_match_opponent_strength`
- `fixture_window_difficulty_score`
- `travel_load_score`
- `rotation_risk_score`

### 14.4. Два специальных риска

#### `pre_big_match_preservation_risk`
Риск того, что команда побережёт силы / состав перед более важным следующим матчем.

Растёт, если:
- следующий матч более важен, чем текущий;
- следующий соперник сильный;
- отдых короткий;
- текущий соперник слабее;
- у команды есть привычка к ротации;
- у команды глубокая скамейка.

#### `post_big_match_relaxation_risk`
Риск спада после эмоционально или физически тяжёлого предыдущего матча.

Растёт, если:
- предыдущий матч был очень важным;
- предыдущий соперник был сильным;
- матч был недавно;
- была высокая интенсивность;
- текущий соперник заметно слабее;
- существует риск недонастроя / отката.

### 14.5. Важная поправка
Календарный фактор не должен автоматически штрафовать любую команду за “3 матча за неделю”.  
Нужно учитывать:
- уровень соперников;
- важность матчей;
- глубину состава;
- стиль тренера;
- домашний/выездной статус;
- travel load.

---

## 15. h2h_context_bias

H2H должен использоваться как слабый contextual bias.

### Разрешённые применения
- tie-breaker между близкими сценариями;
- soft bias в 1X2;
- мягкая корректировка BTTS / over-маркетов.

### Запрещённое применение
- использовать H2H как основной фактор силы команды;
- использовать H2H вместо формы команды.

---

## 16. Итоговая модель для league scorer

Итоговый score команды в матче должен строиться как взвешенная сумма:

- baseline_strength
- current_form
- motivation
- squad_availability
- coach_factor
- schedule_context
- h2h_context_bias

### Концептуально
`team_match_score = weighted_sum(all_factors)`

Итоговые вероятности матча затем строятся через сравнение score хозяев и гостей, плюс:
- correction by draw tendency;
- optional market calibration;
- confidence adjustment.

- schedule context должен быть постоянным существенным фактором, особенно для топ-клубов;
- H2H всегда остаётся слабым.

## 18. Special overrides

Некоторые состояния должны переопределять обычные веса.

### 18.1. Первый матч нового тренера
Если `is_first_match = true`:
- coach factor резко увеличивается;
- старая season form команды частично ослабляется;
- motivation слегка усиливается;
- confidence снижается, если мало данных о предполагаемом составе.

### 18.2. Матчи 2–4 нового тренера
Если `matches_in_charge <= 4`:
- вес `form_under_current_coach` повышается;
- вес старой долгосрочной формы снижается;
- coach factor остаётся повышенным.

### 18.3. Сильный календарный риск перед топ-матчем
Если:
- `next_match_importance = VERY_HIGH`
- и `next_match_opponent_strength = HIGH`
- и `days_to_next_match <= 3`

Тогда:
- schedule penalty усиливается;
- rotation risk растёт;
- confidence в фаворите понижается.

### 18.4. После топ-матча
Если:
- `prev_match_importance = VERY_HIGH`
- и `days_since_last_match <= 3`

Тогда:
- растёт fatigue penalty;
- растёт relaxation risk;
- снижается надёжность обычного фаворитского сценария.

## 19. Логика расчёта календарного риска

Предлагается считать три отдельных score:

### 19.1. `fixture_congestion_score`
Плотность окна матчей:
- сколько игр за последние 14 дней;
- сколько игр в ближайшие 7 дней;
- насколько короткие паузы.

### 19.2. `rotation_risk_score`
Вероятность, что команда выйдет неоптимальным составом:
- следующий матч важнее текущего;
- соперник следующего матча сильнее;
- тренер склонен к ротации;
- есть глубокая скамейка;
- текущий соперник слабее.

### 19.3. `emotional_swing_score`
Риск эмоциональной нестабильности:
- перед матчем сезона;
- после победы над топом;
- после тяжёлого выезда;
- после матча высокой интенсивности.

Итоговый `schedule_context_score` — это комбинация:
- congestion;
- rotation risk;
- emotional swing;
- travel load.

## 20. Правила для экспресса

Экспресс — отдельный режим с повышенной осторожностью.

### 20.1. Почему нельзя брать просто top probability
Некоторые матчи выглядят “хорошо” по базовой силе, но опасны из-за:
- ротации;
- плохого календарного окна;
- слабой определённости состава;
- низкого data confidence;
- эмоционального отката после большого матча.

### 20.2. Вводим express safety class
Каждый матч получает:
- `EXPRESS_SAFE`
- `EXPRESS_CAUTION`
- `EXPRESS_AVOID`

### 20.3. Условия `EXPRESS_AVOID`
Матч должен попадать в `EXPRESS_AVOID`, если выполнено хотя бы одно из условий:

- очень высокий `rotation_risk_score`;
- высокий `pre_big_match_preservation_risk`;
- высокий `post_big_match_relaxation_risk`;
- низкий `starting_xi_confidence`;
- низкий `overall_confidence_score`;
- сильная неопределённость по тренеру / составу;
- фаворит зависит от игроков, чьё участие под вопросом.

### 20.4. Условия `EXPRESS_CAUTION`
- календарный риск есть, но умеренный;
- данные неполные, но не критично;
- матч допустим только при отсутствии более чистых альтернатив.

### 20.5. Условия `EXPRESS_SAFE`
- достаточно высокая вероятность;
- стабильный состав;
- умеренный или низкий календарный риск;
- нет явных признаков ротации;
- хороший overall confidence.

### 20.6. Практическое правило
В режиме экспресса система должна:
1. сначала отсеять `EXPRESS_AVOID`;
2. затем ранжировать `SAFE` и `CAUTION`;
3. только потом собирать комбинацию под целевой коэффициент.

## 21. Confidence framework

Confidence должен считаться не декоративно, а использоваться в логике.

### 21.1. Confidence по блокам
Для каждого блока считается confidence:
- squads;
- coaches;
- schedule;
- news;
- odds;
- teams;
- h2h.

### 21.2. Общий confidence
`overall_confidence_score` должен зависеть от:
- полноты данных;
- свежести данных;
- согласованности источников;
- подтверждённости стартовых условий матча.

### 21.3. Использование confidence
- в обычном прогнозе confidence влияет на уровень уверенности ответа;
- в экспрессе confidence влияет на допуск события в пул.

OpenClaw не принимает решение о прогнозе. Он должен уметь вернуть строго структурированные данные по следующим направлениям:

- дату;

### 22.2. Team context collector
Собирает:
- форма;
- таблица;
- последние матчи;
- календарь;

### 22.3. Squad collector
Собирает:
- вероятные составы;
- травмы;
- дисквалификации;

- риск ротации.

### 22.4. Coach collector
Собирает:

- предыдущие команды;
- историю против соперника и против тренера соперника.

### 22.5. News collector
Собирает:
- важные новости;
- приоритетные сигналы;
- внутрекомандный контекст;
- возможные предпосылки к ротации.

## 26. Финальная философия модели

Эта модель не должна пытаться “угадывать футбол магией”.  
Она должна:

- собирать как можно более качественный контекст;
- честно оценивать неопределённость;
- не переоценивать громкие названия команд;
- быть особенно осторожной с матчами топ-клубов в плотном календаре;
- давать не только прогноз, но и оценку надёжности прогноза.

Главная идея:
**для одиночного прогноза можно позволить себе больше контекстного риска, чем для экспресса.**  
Именно поэтому календарь, ротация, тренерские изменения и подтверждённость состава должны быть не декоративными факторами, а центральной частью логики отбора матчей и расчёта уверенности.
